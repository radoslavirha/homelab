# IoT Overview (Agent Reference)

Focused summary of the IoT layer for agents. Skip to the section relevant to your task.
For infrastructure/deployment details see `AGENTS.md` and `docs/provisioning.md`.

> **Note on target cluster:** IoT services currently run on **server2**. Final destination is **server1** (name TBD). When server1 is ready, IoT stack will migrate there. Until then, server2 is canonical.

---

## Physical Devices

| Device                | IP                                 | Role                                                                                                   |
|-----------------------|------------------------------------|--------------------------------------------------------------------------------------------------------|
| **Loxone Miniserver** | local network (IP not stored here) | Smart home controller — communicates via MQTT through EMQX. Sends state, receives device notifications |

Loxone is the only physical IoT controller. MiOT Bridge supports UDP transport but Loxone integration uses MQTT exclusively.

---

## Services

| Service | Where | Purpose |
|---------|-------|---------|
| **EMQX** | server2, namespace `iot`, `mqtt.server2.home` | MQTT broker. All IoT traffic routes through it. Dashboard on port 18083 (HTTP), MQTT on port 1883 (TCP via Traefik entrypoint `mqtt`) |
| **InfluxDB2** | server2, namespace `iot` | Time-series storage for IoT sensor data. Org: `homelab`. Exposed via HTTPRoute |
| **Telegraf** | server2, namespace `telegraf` | MQTT consumer → InfluxDB2 writer. No inbound ports |
| **miot-bridge-api** | server2, namespaces `production` + `sandbox` | Bridges Loxone ↔ MiOT cloud devices. HTTP + UDP ingress. Polls MiOT devices every 5 s |
| **interactive-map-feeder-api** | server2, namespaces `production` + `sandbox` | Feeds data to an interactive map UI. HTTP ingress only. No secrets |

## Data Flows

### Loxone sensor data → InfluxDB2

```
Loxone Miniserver
  → publishes to EMQX via MQTT
      topic: iot/loxone/state  (JSON: measurement, room, + fields)
        → Telegraf subscribes, parses json_v2
          → writes to InfluxDB2 bucket: loxone
            → InfluxDB2 task "Downsample Loxone" (every 10 min)
              → aggregates mean → bucket: loxone_downsample
```

### MiOT device notifications → Loxone

```
miot-bridge-api polls MiOT cloud (every 5 s)
  → on change: publishes to EMQX
      topic: iot/miot-bridge/device/{deviceId}/notifications  (JSON payload)
        → EMQX rule LOXONE_MIOT_NOTIFICATION fires
          → republishes each JSON property as separate message
              topic: iot/miot-bridge/devices/notifications
              payload: "deviceId=<id>\n<property>=<value>"  (text/plain)
            → Loxone subscribes and reacts
```

## MQTT Topics

| Topic | Publisher | Consumer | Format |
|-------|-----------|----------|--------|
| `iot/loxone/state` | miot-bridge-api (or Loxone directly) | Telegraf | JSON: `{measurement, room, <fields>}` |
| `iot/miot-bridge/device/+/notifications` | miot-bridge-api | EMQX rule | JSON: property→value map |
| `iot/miot-bridge/devices/notifications` | EMQX rule (republish) | Loxone | text/plain: `deviceId=X\nprop=val` |

Production topic prefix: `iot/`
Sandbox topic prefix: `iot/<namespace>/` (namespaced to avoid collisions)

---

## EMQX Rules

### LOXONE_MIOT_NOTIFICATION

- **Trigger:** any message on `iot/miot-bridge/device/+/notifications`
- **What it does:** flattens a JSON object into individual key=value messages, one per property
- **Output topic:** `iot/miot-bridge/devices/notifications`
- **Output payload format:** `text/plain`
  ```
  deviceId=<id from topic segment 4>
  <property>=<value>
  ```
- **Why:** Loxone virtual inputs can only consume simple text/plain MQTT payloads, not JSON objects

Config source: [gitops/helm-values/emqx.yaml](../gitops/helm-values/emqx.yaml) under `emqx.emqxConfig.EMQX_rule_engine__rules__LOXONE_MIOT_NOTIFICATION`

## miot-bridge-api

Custom app bridging Loxone and MiOT cloud devices.

- **Image:** `radoslavirha/miot-bridge:0.16.0`
- **Protocols:** HTTP (port 4000) + UDP (port 4000) — UDP transport supported but Loxone integration uses MQTT
- **Polling:** every 5 s, dispatch on change only, skip 20 cycles after 3 consecutive errors
- **Notifications to Loxone:** via MQTT (EMQX rule reformats and delivers to Loxone)
- **MQTT:** publishes device state to `iot/miot-bridge/device/{deviceId}/notifications`
- **Storage:** MongoDB (database + user auto-provisioned per env)
- **Envs:** production (`udp-miot-prod` entrypoint) + sandbox (`udp-miot-sbx` entrypoint)

Credentials needed:
- EMQX MQTT username/password (provisioned)
- MongoDB database/username/password (provisioned)

## interactive-map-feeder-api

Custom app feeding data to an interactive map UI.

- **Image:** `radoslavirha/interactive-map-feeder:0.6.0`
- **Protocol:** HTTP only (port 4000), 2 replicas
- **No secrets, no MQTT, no UDP**
- **Envs:** production + sandbox

## Telegraf Data Pipeline

- **Input:** MQTT `iot/loxone/state` — JSON parsed with `json_v2`, extracts `measurement` as measurement name, `room` as tag, all other fields as fields
- **Processing:** maps `status` field string → numeric code (`healthy=1`, `problem=2`, `critical=3`)
- **Output:** InfluxDB2 bucket `loxone`, org `homelab`, filtered by tag `source=loxone`
- **Auth:** write-only token from OpenBao `secret/server2/telegraf-influxdb2` (provisioned by influxdb2 provisioner job `telegraf`)
